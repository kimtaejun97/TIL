## ? JDBC

애플리케이션 서버와 DB의 일반적인 연동은 다음과 같다.
1. TCP/IP를 이용해 커넥션을 연결한다.
2. DB가 이해할 수 있는 SQL을 커넥션을 통해 DB에 전달한다.
3. DB는 SQL을 수행하고 결과를 응답한다.

그런데 여기서 문제가 발생한다. DB의 종류는 수십개에 달하고, 데이터베이스마다 커넥션의 연결방법, SQL 전달방법, 결과의 응답이 모두 다르다는 것이다.    
이를 해결하기 위해 표준이 필요했고 JDBC가 등장하게 되었다.

### JDBC 표준 인터페이스(Java Database Connectivity)
자바에서 연결, SQL 전달, 결과 응답에 대한 표준 인터페이스를 정의하여 제공하였고, 각 DB 벤더들은 자신의 DB에 맞게 이를 구현하여 라이브러리 형태로 제공하게 된다.   
(MySQL JDBC 드라이버, Oracle JDBC 드라이버 ...)   
이러한 구조는 다른 코드의 변경없이 JDBC 드라이버만 변경하여 다른 DB를 사용할 수 있게 해준다.   

그러나 데이터베이스마다 SQL의 문법이나 데이터타입이 다른 부분이 있는데, 그 부분은 해결해 주지 못해 SQL은 데이터베이스에 맞게 변경해 주어야 한다.    
최근에는 JDBC를 직접 사용하기 보다는 SQL Mapper, ORM 기술로 조금 더 편리하게 사용한다.(JDBC를 직접사용하면 복잡)


#### SQL Mapper
JDBC Template, MyBatis 가 대표적이며, JDBC의 반복 코드를 제거해주고, 응답을 객체로 편리하게 변환해준다.    
#### ORM
표준 인터페이스인 JPA가 있고, 이를 구현한 하이버네이트, 이클립스링크 등의 구현체가 있다.   
관계형 데이터베이스의 테이블과 객체를 매핑해준다.       
ORM 기술이 개발자 대신 SQL을 동적으로 만들어 실행해주기 때문에 반복적인 코드의 작성을 줄일 수 있으며 데이터베이스간의 문법이달라 발생했던 문제들도 중간에서 해결해준다.   
SQL을 작성하지 않아도 되기 때문에 생산성이 높아지지만, 실무에서 사용하려면 깊이 있는 학습이 필요하다.
